Optimizing for Almost Sorted Data: Sort Pushdown in Apache DataFusion (5 minute read)

Apache DataFusion now exploits sortedness automatically, even when data is only partially sorted or its ordering was never declared. The engine discovers a global sort order from per-file min/max statistics, deletes redundant sorts at plan time, and biases scan order toward the most-promising files, with dynamic filters driving row-group pruning at runtime. The payoff is ORDER BY ... LIMIT queries that skip the blocking full sort and full scan, cutting latency and peak memory on large tables.

Why A Frontier Data Agent Outperforms General Coding Agents in Quality and Cost (1 minute read)

Databricks benchmarked its Genie Code data agent against three general-purpose coding agents on 401 real internal tasks, each capped at a 20-minute budget, and found Genie Code was both the most accurate and the most cost efficient. The argument is that data agents can't lean on the verifiable tests coding agents use, so semantic catalog search, persistent memory of trusted tables, and enterprise context let Genie Code skip the token-burning random-walk exploration that sinks general agents.

Aurora DSQL: Scalable, Multi-Region OLTP (5 minute read)

Amazon's engineers detail Aurora DSQL, a distributed PostgreSQL-compatible OLTP database that disaggregates compute, transaction commit, and storage into independently scalable layers with active-active multi-region writes. The design leans on optimistic concurrency control and precise EC2 clock synchronization to guarantee strong consistency without a two-phase commit coordinator on the hot path. The bet is that decoupling the write path from storage lets a single database scale horizontally while surviving regional failure, a trade-off that swaps lock-based contention for commit-time conflict aborts.

DuckDB Internals: Why is DuckDB Fast? (Part 2 Vectorized Execution) (5 minute read)

DuckDB's second internals post explains vectorized execution: instead of processing one row at a time, DuckDB pushes batches of values through each operator, keeping the CPU fed rather than stalling on memory waits and instruction dependencies. The payoff is concrete. The 2005 MonetDB/X100 benchmarks it cites clocked MySQL's row-at-a-time engine at 0.8 IPC and ~49 cycles per addition, versus ~2.2 cycles for the batched approach that DuckDB now builds on.

Benchmarking the Full Pipeline of Materialized-View-Based Query Rewriting (1 minute read)

This paper benchmarks materialized-view query acceleration as a full pipeline: candidate enumeration, view selection under storage budgets, and optimizer rewriting, rather than studying one stage in isolation. Using a modular framework with controlled ablations and a cross-engine protocol, the authors find strong interaction effects across stages and wide variability in MV usage and realized savings across academic methods and open-source and commercial systems. They also catalog recurring failure modes that explain regressions after rewriting.

If You Think You Can Do Real-World Text-to-SQL (5 minute read)

The author argues that benchmark scores like those on Spider and BIRD wildly overstate how well text-to-SQL works against real production databases, where schemas are sprawling and undocumented, column names are cryptic, and business questions are ambiguous. The post walks through why enterprise schemas with hundreds of joined tables, inconsistent data, and implicit domain rules break models that ace academic benchmarks. The takeaway for data teams is that the hard part is schema semantics and context, not SQL generation itself.

Demonstrating GenDB: Instance-Optimized and Customized Query Processing Code Generation via LLM Agents (1 minute read)

GenDB is a generative query engine that uses LLM agents to write instance-optimized query execution code tailored to specific data, workloads, and hardware, rather than relying on a hand-built engine. It targets repetitive, templated queries where upfront generation cost amortizes across runs, pairing with a traditional DBMS for ad-hoc queries. The prototype beats state-of-the-art engines on TPC-H and a leakage-resistant benchmark, with correctness enforced through fuzz testing.

Worst-Case Optimal BGPs on Temporal Graphs (1 minute read)

This paper presents an index that evaluates basic graph patterns over temporal graphs, where each edge carries an interval of validity, in worst-case-optimal time using O(N) space for N edges. The authors extend Leapfrog Triejoin to handle time-aware quads (subject, property, object, time) under any variable ordering, and get the same guarantees for snapshot, version, and other temporal query variants. On real datasets, realistic queries return in milliseconds with low space overhead.

Auto-Fill: Learning to Predict Missing Values Accurately with Specialist Language Models (1 minute read)

This paper introduces Auto-Fill, an approach to predicting missing cell values in tabular data that post-trains three specialist small language models, each optimized for one capability: world knowledge, text-based reasoning, or code-based reasoning. A calibrated ensemble picks the most confident specialist or abstains, cutting the hallucinated predictions that make frontier reasoning models unreliable for data cleaning. Across 11 benchmarks and 2,200 real tables, it beat o3-pro, Gemini 3 Pro, and DeepSeek R1 on accuracy at under 1% of their cost.

RCC: Speculative Write Versioning with Redo Logs (1 minute read)

This paper introduces RCC, a concurrency control scheme that uses redo logs to let write-write conflicting transactions run speculatively instead of serializing behind each other. Each transaction creates out-of-place speculative versions and installs them only at commit, with commit-time deadlock detection and column-granule logging to avoid false conflicts. On MySQL and PostgreSQL, RCC improves TPC-C throughput and latency by an order of magnitude at 64 threads, and scales to 128 clients on high-contention YCSB where competing schemes stall past 32.

Backfilling: the most underrated feature of a data pipeline (5 minute read)

Backfilling gets treated as a throwaway migration script written under pressure, then abandoned in a scripts/ folder to rot. The author argues both forms of backfill, historical ingestion and system-to-system migration, are the same problem and belong inside the production extract-raw-transform pipeline rather than one-off jobs. Running historical loads through the same tested stages as tomorrow's data cuts the time teams spend fighting their own pipeline.

Smart caching with CTE on Spark (5 minute read)

Caching in Spark speeds up repeated computations by holding intermediate results in memory, but this piece argues the win only materializes for medium-sized, frequently reused datasets. Tiny tables gain nothing, and tables too large for memory trigger disk spills or eviction that slow the job down. The practical guidance for CTEs and nested queries: cache only when reuse is guaranteed, and watch the Spark UI storage tab to confirm the cache is actually paying off.

DMG: A Scalable and Efficient Memory-Disaggregated Graph Processing System (1 minute read)

DMG is the first graph processing system on disaggregated memory that scales past a single compute and memory node, sidestepping the oversized compute-side caches prior DM designs required. It pairs a DM-friendly graph store with an adaptive update coordinator and a two-stage workload manager for partitioning and runtime rescheduling. Against the prior state-of-the-art, the authors report up to 4.9X better performance while cutting compute-side cache demand by up to 18.9X.
