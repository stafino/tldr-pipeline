How to achieve pruning when querying by non-partitioned columns in PostgreSQL (5 minute read)

This post shows how to get PostgreSQL partition pruning when filtering by non-partition-key columns, using an events table partitioned by timestamp but queried by session_id. The author leverages correlations in the data to derive an implicit timestamp predicate from the queried column, letting the planner eliminate entire partitions instead of scanning them all.

Better tools made Copilot code review worse. Here’s how we actually improved it. (1 minute read)

GitHub found that swapping Copilot code review onto the shared Copilot CLI tools (grep, glob, view) raised review cost and caught fewer issues, because the tools' instructions assumed the sparse tool-calling behavior of older models. Rewriting the instructions to match how a reviewer actually reads a pull request flipped the regression into roughly 20% lower average review cost at the same review quality.

In defense of not understanding your codebase (3 minute read)

The author argues that fully understanding your codebase is only realistic on small, low-turnover projects, and that on large systems like Google search or GitHub, partial understanding of your local area is not a failure but the correct default. He pushes back on Peter Naur's "programming as theory building," rejecting Naur's claim that a system should be discarded and rebuilt from scratch once the original team's mental model is lost.

In Emacs, everything looks like a service (1 minute read)

Charles Choi argues that Emacs is best understood not as an operating system but as a universal client, using its built-in access to system services, networking, and storage to improvise client behavior for almost any task. The post walks through the client-server model and Emacs' built-in libraries for UI, network edge, and local databases like SQLite, making the case that once inside Emacs, every resource reduces to a service you can build a client against.

GPT-5.6, Grok 4.5, Claude, and Muse Spark build the same 4 apps (1 minute read)

Muse Spark 1.1 as comparison points. The author reran their model build-off with twelve models generating a raycaster, Rubik's cube, calculator, and Game of Life, five attempts each, publishing every artifact with cost and latency instead of a scored verdict. Their takeaway is that run-to-run variance is large enough that single-shot comparisons mislead, so they link all raw attempts and let readers judge which model actually built something playable.

Write code like a human will maintain it (1 minute read)

The author argues that letting an LLM copy-paste the same access-check conditional across route handlers, jobs, and webhooks isn't a harmless shortcut, because the model reads your existing code and treats every merged duplication as the house style. The next request builds on the four copies already in the repo, so code smells compound into a pattern the model actively preserves through refactors instead of fixing.
